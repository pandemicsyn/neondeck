Functional research · live disposable PR

Autopilot worked, but acceptance is not complete.

This report records a live end-to-end test of Neondeck PR Autopilot against a disposable GitHub pull request, using an isolated runtime home and the configured Kilo provider.

PR #176 closed · unmerged isolated runtime 2026-07-21 UTC
Assessment

The core owner, approval, safe-push, idempotency, restart, and cleanup paths are real. Four behavioral defects and one configuration-chat reliability problem should be fixed before calling the feature fully production-ready.

Direct answer

How the first change was approved

Approval was sent through the Active Watches UI.

After the watch reached waiting, I used the watch card’s embedded owner-message input and sent: “Approved. Push exactly the prepared commit and post a concise result.”

That input talks directly to the persistent pr-autopilot-owner instance. It was not sent through the general Neon/display-assistant chat, and it was not a GitHub “Approve” review. The direct human message was the authority grant for that turn, which is when the owner received push/respond tools.

GitHub inline review comment → watch detects actionable feedback → persistent owner edits and commits → watch becomes waiting → human reviews diff in Active Watches → human sends “Approved…” to owner input → approval turn gains push/respond tools → exact prepared SHA is pushed

Method

Test execution overview

Preflight and disposable PR creation

Started Neondeck from an isolated home, verified provider dispatch, created a harmless Markdown fixture, pushed a temporary branch, and opened PR pandemicsyn/neondeck#176. The PR was explicitly marked disposable and never merged.

Passive watch and lazy ownership

Created a notify-only watch through the dashboard chat’s typed configure action. Confirmed there was initially no owner or managed worktree. Increased authority through the Active Watches mode selector and its explicit confirmation.

Approval-mode change

Added an inline review request changing approval: pending to approval: handled. The first actionable event lazily bound owner pr-owner-f667253464e4ff827bafdbb0 and worktree 908e7649-0976-48a4-aac2-fcd945675c62. The owner prepared commit ee595bb0978794e9c82edda3bfb5419f675efa5d without push authority.

Restart and direct-human approval

Restarted Neondeck while the watch was waiting. The same owner, worktree, diff, commit, and conversation recovered. Reviewed the diff in Active Watches, then approved through its direct owner input. The exact prepared SHA was pushed and one concise result comment was posted.

Autonomous fail-closed path

Increased the mode to autofix-push-when-safe, paused polling, and added a request changing autonomous: pending to autonomous: handled. Nothing ran while paused. After resume, the owner prepared commit 6938fa14f46e538935eea6d2e62c14593f5798bd and correctly refused to push because no required check was configured.

Policy repair, credential proof, and safe push

Configured and preapproved the exact targeted check git diff --check HEAD^ HEAD -- AUTOPILOT_FUNCTIONAL_TEST.md. The check passed. SSH then failed closed because the transport actor could not be proven to match the GitHub API actor. A process-only HTTPS URL rewrite provided verifiable credential identity without changing shared Git configuration. A fresh review-thread instruction caused safe-push to re-run and push the exact prepared SHA non-force.

Idempotency, terminal state, and cleanup

Forced another scheduler tick and observed no duplicate commit, push, or PR comment. Closed the PR, observed complete, confirmed direct owner messages were rejected, and verified the managed worktree was deleted while the durable owner conversation remained readable. Finally deleted the remote branch and all temporary runtime resources.

Findings

Defects to address

P1

Completed watches can be resumed

After terminal cleanup, the dashboard still showed resume. The control API accepted it and re-enabled the scheduled task even though autopilotStatus=complete. The test immediately paused it again. Terminal watches should reject pause/resume/retry mutations, and the UI should hide or disable those controls.

P1

Chat configuration does not behave like a bounded typed mutation

The display assistant was asked to make one exact policy change. It inspected unrelated files and shell state, called multiple tools, duplicated commands, and changed guardrails and mode fields that were supposed to remain untouched. The deterministic actions were sound when invoked directly; the chat entry point was not. Configuration prompts need stronger tool routing and patch/field-preservation semantics.

P2

Retry does not guarantee a fresh safe-push probe

After changing the Git environment from unverifiable SSH to verifiable HTTPS, a plain retry delivered a new owner turn, but the persistent model reused its previous failure conclusion and did not call safe-push again. A new review-thread message was required to force reevaluation. Retry should carry an explicit “re-run current gates now” instruction or deterministically invoke the safe-push attempt outside model discretion.

P2

Autonomous success omitted the PR response

The autonomous owner called safe-push successfully and reported success in its conversation, but never called its response tool. The PR therefore had no concise autonomous-result comment. Successful autonomous completion should deterministically require or perform the result response.

P3

Outdated review comments are redelivered as new

When a push changed line/outdated state, feedback fingerprints changed and earlier comments were emitted again as new actionable deltas. The owner no-op’d them, so external effects remained idempotent, but this consumed model turns and generated noisy notifications. Delivery identity should remain stable across positional/outdated metadata changes.

Architecture assessment

What the run validated

The simple model is real

One watched PR retained one stable Flue instance and one managed worktree. The owner’s canonical conversation recorded feedback, tool calls, approval, failures, retries, and the successful push. This matches Flue’s durable-agent model: later turns reconstruct context from one continuing stream rather than inventing workflow-per-transition state.

Capability ceilings held

The review-event turn in approval mode could edit and commit but could not push. The direct-human approval turn gained push/respond tools. Autonomous mode received push only behind targeted checks, current-head verification, policy, credential, and non-force gates. Both missing checks and uncertain identity failed closed.

Bottom line

The implementation is not an elaborate distributed coordinator; the tested core is the intended small watch/owner/worktree loop. The remaining problems are boundary and instruction-following defects, not evidence that the core architecture needs more machinery. Fix the deterministic terminal/retry/response boundaries and tighten configuration routing before adding new coordination concepts.
