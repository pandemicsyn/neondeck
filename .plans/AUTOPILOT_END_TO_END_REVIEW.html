Verdict: not usable as an autopilot

The automatic watched-PR path ends after creating a managed worktree. It never spawns or briefs a coding agent, generates a fix, verifies it, requests/uses approval, pushes, comments, or cleans up. Today this is a notification and checkout-preparation foundation with individually implemented downstream primitives.

Executive summary

The product promise and the implementation do not meet

The README promises that watching a PR lets Autopilot prepare a fix, run checks, hand over a diff, or push when policy allows. The roadmap marks the PR event Autopilot and policy/UX phases complete. The production coordinator, however, implements only watch delta → triage → prepare-pr-worktree. Its durable admission state machine ends at prepared, where “prepared” means only that a checkout exists; it does not mean that a diff exists.

The deterministic review and CI fixers are useful services, but they require a caller-authored patch or replacement plan. No model turn, Neon coding agent, delegated worker, scheduler transition, or user surface supplies that plan for watched feedback. The automatic path therefore cannot produce the feature’s core outcome.

Functional journey Critical

Stops before any code fix is generated.

Safety architecture Strong

Typed policy, isolation, locks, approvals, and push gates.

Setup & discoverability Poor

Hidden mode setup; new installs default to notify-only.

Verification confidence Partial

Strong unit fixtures; missing product-path coverage.

Release recommendation

Do not present the current behavior as autonomous fixing. Either ship the missing coordinator/planner and close the activation, readiness, and operator gaps below, or temporarily rename the current feature to “PR watch + managed worktree preparation” and describe later stages as manual primitives.

Behind the scenes

The actual production flow

Entry Watch created

Chat, dashboard, CLI, or local API writes a PR watch and an interval scheduled task. Mode is configured separately.

Poll Scheduler polls

Basic PR state refresh runs, then GitHub event state is fetched into per-category watermarks.

Delta Deltas classified

Review threads, requested changes, checks, commits, mergeability, and freshness become structured deltas.

Admission Admission claimed

SQLite deduplicates by watch/event fingerprint and applies global, repo, and same-PR concurrency limits.

Triage Flue triage runs

triage-pr-event deterministically chooses no-op, explain, prepare, approval, or safe-push classification.

Checkout Worktree prepared

prepare-pr-worktree gathers facts and attempts an isolated detached checkout. No patch is generated.

Dead end Automatic path ends

Admission becomes prepared. There is no transition to fix, verify, approval, push, comment, or cleanup.

Manual only Standalone primitives

Fix, verify, push, comment, prepared-diff, and recovery services exist, but require explicit/manual/raw-API initiation.

Durable state involved

The flow spans PR watches and scheduled tasks, pr_watch_event_watermarks, autopilot_admissions, Flue run observations/events, worktrees and worktree events, notifications, and, only if a disconnected fixer is called, prepared diffs, approvals, and verification/push summaries.

User interaction review

How a user can and cannot use it today

Register a local repository and GitHub token

neondeck init collects a checkout and GITHUB_TOKEN. It validates identity, not full Autopilot read/write/git readiness.

Available, incomplete readiness

Enable a mode

The fresh default is notify-only. The only typed mutation is a per-repo config action/local API. There is no onboarding step, settings UI, CLI mode command, or /autopilot command.

Hidden; separate from watch creation

Create a PR watch

The user can ask Neon, use /watch-pr, click Watch in the GitHub panel, run neondeck watch-pr, or call POST /api/watches. None accept a mode.

Available; silently inherits policy

Process current feedback

There is no “process existing feedback now” option. The first event poll seeds a baseline and suppresses all current actionable state.

Unavailable

Observe Autopilot

The Autopilot panel can show policy, queue-like state, prepared diffs, approvals, checks, and activity. Recommended fresh dashboard presets omit that panel, so notification deep-links can fail.

Implemented, hidden by onboarding

Ask Neon to fix review feedback

The review fixer action rejects interactive chat and has no dashboard/CLI trigger. Its raw API requires the caller to author exact replacements or a patch.

No normal user path

Approve, verify, and push a prepared diff

These controls work only after some separate path creates a prepared diff. The automatic watcher never reaches this state. Prepare-only diffs still receive push approvals that cannot succeed under their mode.

Downstream primitive only

Pause or stop

Pause controls polling only. Stop deletes the watch/task but does not cancel admitted runs, abandon diffs, remove worktrees, or purge admissions. The confirmation copy does not explain that distinction.

Control semantics are misleading

Natural-language setup does not have a first-class contract

The display assistant technically has both neondeck_watch_pr_add and neondeck_config_update_repo_autopilot_policy, so a sufficiently explicit conversation may issue two separate action calls. There is no composite, tested “watch this PR in mode X” action, and no agent guidance that defines the two-step setup, confirmation, baseline handling, or expected outcome. Asking “watch this PR and keep it moving” normally maps to a watch that remains notify-only.

Severity-ranked findings

Bugs, workflow gaps, and UX defects

F1 Critical

No agent/planner or downstream coordinator exists

Watched review feedback and CI failures can reach triage and worktree preparation, but nothing invokes a coding agent or continues through fixer, verification, approval/push, comment, or cleanup. The fixer workflows are thin wrappers over deterministic actions and require caller-supplied replacements or patches. Without edits, no fix is applied.

User impact: the core promise, “watch this PR and keep it moving,” cannot complete. There is no worker to brief about the checkout, GitHub, branch, tools, checks, or authority because no worker is spawned.

F2 High

Existing feedback and the watch-creation race are discarded

Watch creation does not seed event watermarks. The first scheduler event refresh sees no previous watermarks, stores the current state as a baseline, and returns without triage, even if the PR already has failing CI, requested changes, or unresolved threads. Feedback arriving between watch creation and the first poll is lost the same way.

Manual event refresh only refreshes state; it does not run the scheduler admission logic. The user has no “process current feedback now” control.

  2. Wait for the first scheduled poll.

  4. No action occurs until a later watermark changes.

F3 High

Fresh or fork PR worktrees can fail before fetch

Preparation calls git worktree add --detach ... <headSha> before the optional sync/fetch step. The scheduler does not pass fetch: true, and even if it did, the fetch would happen after creation. A common newest PR/fork SHA that is not already present locally cannot be checked out.

Required correction: fetch the exact PR head ref/SHA into the source repo before worktree creation, with fork-aware remotes and a bounded failure message.

Approval, safe-push, and result-comment semantics are internally inconsistent

Every prepared diff creates a pending push approval, including prepare-only. The dashboard’s “approve” action defaults to verify-then-push, but the push gate later rejects prepare-only mode. Agent briefing, environment, and permissions

The expected continuing PR-owner session and briefing do not exist

No coding worker is spawned in the watched-PR path. The configured Flue workflows name displayAssistant as their agent but execute deterministic actions directly; they do not start a model planning turn. The display assistant’s own working directory is the virtual /workspace, explicitly not the host checkout. Its available subagents are research/CI/release analysis profiles that are instructed not to discover or run host commands.

Target lifecycle: one Neon session and workspace per watched PR

The first actionable event should lazily create one narrowly scoped PR-owner Neon session and one managed worktree. Later feedback, CI failures, and PR state changes should be delivered as fresh authoritative event envelopes to that same session, with same-PR turns serialized by the durable controller. A routine event must not create another Neon session. The owner session and workspace remain until the PR reaches its configured terminal state or the user explicitly stops Autopilot; exceptional session rotation is explicit, audited, and carries forward a compact handoff.

A good existing pattern is available

The explicit dashboard Fix CI path is separate from watched Autopilot but does create/sync a managed worktree and start Kilo in that real directory. Its handoff includes the workspace path and managed-worktree status. It still needs a verified tool/auth capability inventory, but it is a useful model for the missing autonomous worker contract.

Architecture to retain

The foundations are valuable

The failure is primarily missing orchestration, planning, activation, and product closure, not an absence of safety infrastructure. The following pieces should be preserved and composed into the finished loop:

Deterministic facts

GitHub event model and watermarks

Event collection is structured, persistent, and quiet when nothing meaningful changes. Deltas cover commits, inline review threads, requested changes, checks, mergeability, and freshness.

Concurrency

Atomic admission and deduplication

Claims are idempotent by watch/event fingerprint and enforce global, per-repo, and same-PR mutation capacity.

Isolation

Managed worktrees and repo-edit boundaries

Code-changing work is designed to stay out of the user’s primary checkout, with locks, dirty-state checks, path containment, and guardrails.

Push safety

Strong deterministic gates

Push checks mode/policy, SHA- and policy-bound approval, verification against current HEAD, branch permission facts, clean committed state, and forbids force push.

Operator review

Prepared diff and recovery services

Diff inspection, verification records, revisions, retry, resync, cleanup, and result notifications provide a solid backend for one web/TUI runtime.

Explicit worker path

Bounded Fix CI / Kilo safeguards

The separate human-triggered CI path rejects partial facts and head movement, uses an exact managed worktree, and prohibits implicit push/comment effects.

Recommended implementation sequence

Close the loop in five gates

Implementation source of truth

The executable work packages, state model, PR-owner session/event briefing, verification matrix, and package exit gates live in AUTOPILOT_IMPLEMENTATION_PLAN.md.

Gate 1 · Core coordinator

Add a durable planner/fixer stage

  • Extend admissions through plan/fix, verify, approval, push, comment, cleanup.
  • Create/reuse one continuing PR-owner Neon session; bound and audit each event turn.
  • Feed typed patch/replacement output into existing deterministic fixers.
  • Branch correctly by mode; never create a push approval in prepare-only.
Gate 2 · Activation

Create one first-class setup operation

  • Add autopilot_watch_pr_configure(ref, mode, processExisting).
  • Expose it in chat, dashboard, CLI, and docs.
  • Put Autopilot in recommended dashboard presets and onboarding.
  • Make current mode and polling state visible beside each watch.
Gate 3 · Readiness

Preflight the real environment

  • Fetch the PR head before worktree creation.
  • Validate full feedback availability and token scopes.
  • Verify noninteractive git credentials and commit identity.
  • Resolve required checks and unattended preapproval before admission.
Gate 4 · Durable recovery

Make work cancellable and finite

  • Expire attached but lost/stuck runs.
  • Add exponential backoff, attempt caps, and explicit resume.
  • Separate pause watch, cancel turn, abandon diff, archive session, and cleanup worktree.
  • Archive terminal admissions and purge/supersede on watch removal.
Gate 5 · Product verification

Test the user promise, not the primitives alone

  • Drive a scheduled watch from baseline through a new review delta.
  • Use a real Flue session-dispatch boundary and a continuing fake planning agent.
  • Assert diff, checks, approval/push, comment, notifications, and cleanup.
  • Test restarts, duplicates, stale SHAs, forks, permissions, and cancellation.
Documentation

Align claims with shipped behavior

  • Publish prerequisites, modes, trust boundaries, and exact setup recipes.
  • Document initial baseline/process-existing behavior.
  • Explain approval side effects, stop semantics, and recovery.
  • Mark the roadmap complete only after acceptance tests pass.

Definition of usable

Required end-to-end acceptance scenarios

  1. In chat, “watch owner/repo#123 in prepare-only and process current feedback” creates one watch, persists the PR override, confirms any authority increase, and reports exact next poll/current processing state.
  2. A PR that already has unresolved inline feedback at activation is admitted once when processExisting is requested; safe baseline-only behavior is also selectable and clearly explained.
  3. The first actionable review event starts one PR-owner Neon session with the exact managed worktree path, repo/PR/branch/SHAs, full feedback, available tools, checks, and policy limits. Later feedback is injected into that same session and workspace with refreshed, authoritative facts.
  4. Overall requested-changes review bodies and supported PR conversation comments retain their text and can trigger a fix or an explicit “unsupported/manual” outcome.
  5. A new or fork PR head absent from the local clone is fetched before checkout and produces a usable worktree without touching the primary checkout.
  6. Prepare-only produces a real inspectable diff, never creates a push approval, and stops with review/revise/abandon/cleanup actions.
  7. Autofix-with-approval produces and verifies a diff, explains the effect of approval, pushes only after explicit approval, then posts one fact-grounded result comment.
  8. Safe-push mode automatically verifies, checks credentials and branch permissions, pushes without force, posts the result, and schedules cleanup only when every gate passes.
  9. Missing token scopes, git credentials, commit identity, required check preapproval, or model provider blocks before mutation with a precise setup instruction and no retry storm.
  10. Pause, stop watching, cancel active work, abandon a diff, and clean a worktree are distinct controls with accurate confirmation copy and durable audit state.
  11. Restart during every stage reconciles exactly once; attached lost runs time out; repeated failures back off and cap; same-PR changes serialize without permanently consuming capacity.
  12. The dashboard shows one canonical row per PR/task with stage, mode, reason, timestamp, run link, and available action; it never hides an approval due to client-side slicing.
  13. A fixture-backed product test executes watch delta → triage → continuing owner-session turn → prepared diff → verification → approval/push → later feedback in the same session → terminal archive/cleanup, and the smoke command actually runs it.
