# Scheduler, Routines, and Autopilot Mechanics Review

Date: 2026-07-09
Baseline: `origin/main` at `7ec91b96` (`Polish repository README`)
Method: static inspection only; no tests, builds, linters, migrations, or app commands were run.

Implementation status (2026-07-10): the findings have been split into four
stacked implementation PRs. The current architecture uses one SQLite
scheduled-task model, durable autopilot admissions, SHA/policy-bound prepared
diff approvals, and a fresh pre-1.0 app-database baseline. This review remains
the historical rationale; the wiring plan and roadmap describe the current
contract.

## Scope and correction

This review covers the current scheduler, watches, routines, PR-event dispatch, autopilot policy, worktree, prepared-diff, approval, and operator-state paths on the fetched `origin/main` tip.

An earlier pass was made against detached commit `99627b53`, four commits behind `origin/main`. That pass is superseded by this document. Current `main` includes the scheduler PR-event dispatch extraction and earlier autopilot feedback-loop wiring. In particular:

- scheduler job failures are now contained per job rather than aborting the whole tick;
- PR-watch jobs now refresh detailed event watermarks and automatically admit `triage-pr-event` for meaningful deltas when effective policy is not `notify-only`;
- blocked or failed triage admission is retained in the scheduler job result and retried or superseded on a later tick.

Those are material improvements and should not be reported as missing.

## Executive summary

The scheduler foundation is generally careful: it has process and SQLite leases, advances jobs before external effects to reduce duplicate admission, records Flue scheduler runs, contains individual job failures, and gives the newer routines subsystem transactional single-flight and concurrency behavior.

The implementation is nevertheless carrying two competing, unshipped scheduling models:

1. `schedules.json` entries compiled into interval-based `jobs` rows; and
2. SQLite-backed routines with interval, once, and a custom five-field UTC cron implementation.

There is no production compatibility requirement for either representation. The `schedules.json` path should be deleted, not migrated or retained behind aliases. Neondeck should have one SQLite timing/admission mechanism with three explicit task kinds: poll a PR watch, run a briefing, or execute a user-authored agent instruction.

Flue already defines the execution model for scheduled work: each occurrence should start a bounded workflow or send an event to a continuing agent. On the installed Node target, Flue intentionally delegates clock/cron scheduling to the host application. Neondeck therefore still needs one small timing and durable-claim layer, but it should hand every agent/workflow occurrence to Flue instead of implementing a parallel execution model.

Autopilot is now automatically ignited through watcher deltas, but it is not yet an end-to-end autonomous loop. The scheduler admits triage; triage classifies the event and returns `nextWorkflow`, but no production coordinator consumes that result to prepare a worktree, fix, verify, push, or comment. The current operator “queue” is still derived from watches, worktrees, approvals, prepared diffs, notifications, and Flue observations instead of representing durable admissions.

The recommended direction is to simplify around two shared primitives:

- one SQLite-backed scheduler for every recurring or one-shot task;
- one durable autopilot admission state machine for watcher-driven PR work.

## Product requirements clarified

The scheduler only needs to support these three use cases:

1. periodically poll a watched PR, detect meaningful changes, and emit autopilot events;
2. run periodic briefings;
3. let a user instruct an agent to perform bounded work on a schedule, for example, “every 12 hours, use CLI `xyz` to do `abc`.”

There is no legacy-data, migration, or backwards-compatibility requirement before 1.0. Code and configuration that do not serve one of these cases should be removed rather than generalized.

## Flue version prerequisite

As of this review, npm's `latest` release line is Flue `1.0.0-beta.9`. The repository already declares `@flue/runtime`, `@flue/sdk`, `@flue/react`, and `@flue/cli` at `^1.0.0-beta.9`, and `package-lock.json` resolves that release family.

Treat that observation as dated, not permanent. Before changing scheduler or autopilot architecture:

1. query the npm `latest` dist-tag for all four Flue packages;
2. upgrade runtime, SDK, React bindings, and CLI together to the versions published under that latest release, respecting their declared peer-version constraints;
3. while Flue remains prerelease, pin the selected versions exactly rather than using caret ranges;
4. regenerate and commit the lockfile in the upgrade change;
5. read the release notes and current Flue schedule, workflow, agent dispatch, Node persistence, routing, and run-inspection documentation;

7. base the cleanup on the APIs and guarantees of the newly installed version, not on `beta.9` assumptions preserved in this review.

Do not combine an unknown Flue-version jump and the scheduler rewrite into one opaque patch. Land or at least verify the framework upgrade as the first isolated cleanup step so API changes, persistence changes, and scheduler changes have distinct failure boundaries.

## Findings

### 1. [P1] Watcher-driven autopilot stops after triage

Current `main` correctly connects PR watches to event watermarks and triage:

- `src/modules/scheduler/dispatch.ts` sends `watch-pr` work through `refreshWatchJob`;
- `src/modules/scheduler/pr-watch-events.ts` refreshes event state, computes deltas, resolves effective watch policy, checks concurrency, and invokes `triage-pr-event`;
- failed or blocked triage inputs are retained in the prior job result for retry.

However, `src/modules/autopilot/triage.ts` only returns:

```text
classification
shouldPrepareWorktree
nextWorkflow: prepare_pr_worktree | null
```

The `triage-pr-event` Flue workflow is a thin action wrapper. No production observer or coordinator consumes the terminal triage result and admits `prepare-pr-worktree`, a fixer, verification, push, or result comment.

Impact:

- `draft-fix`, `auto-fix-no-push`, and `auto-fix-push-after-checks` change automatic triage classification but do not produce an automatic fix;
- operators can drive the later actions through chat/API/dashboard, but the watcher cannot complete the advertised loop by itself;
- the roadmap status for Phase 19 reads more complete than the production admission path.

Recommended fix:

- persist a first-class autopilot admission before invoking triage;
- observe the terminal triage result and advance that admission through explicit states;
- invoke one bounded Flue workflow per state transition;
- use the persisted admission, not a returned `nextWorkflow` string, as the source of truth.

### 2. [P1] Generic repo metadata updates can increase autonomy without confirmation

`src/modules/config/schemas.ts` accepts arbitrary `metadata` on repo add/update. `src/modules/config/mutations/repos.ts` replaces repo metadata without comparing old and new autopilot authority or requiring `confirm=true`.

Because effective repo/watch policy is read from `metadata.autopilot`, a model-callable `neondeck_config_update_repo` action can set or increase:

- repo autopilot mode;
- concurrency or policy limits;
- watch overrides;
- push-related policy fields nested under autopilot metadata.

The roadmap explicitly requires confirmation when enabling or increasing autopilot authority or changing push behavior.

Recommended fix:

- remove `metadata.autopilot` mutation from the generic repo update action;
- add a typed autopilot-policy mutation action with a stable schema;
- rank modes by authority and require confirmation for every upward transition;
- require confirmation when limits are relaxed, push destinations expand, or force-push policy changes;
- apply the same confirmation rule to repo path changes, which also alter the execution trust boundary.

### 3. [P1] `preparePrWorktree` holds a default lock across workflow boundaries

`src/modules/autopilot/worktree.ts` defaults `input.lock` to true, acquires a PR-scoped lock for 30 minutes, returns the lock, and does not release it before the workflow ends.

The next fixer or verifier performs its own concurrency check and tries to acquire its own PR lock. The prepared worktree is therefore visible as busy and the intended next workflow can be rejected until the original lock expires or is manually released. Existing smoke paths commonly pass `lock: false`, which avoids the default production behavior rather than validating it.

Recommended fix:

- make preparation lock-free by default; or
- acquire the lock only around create/sync/inspect and release it in `finally`; or
- if preparation and fixing must be one critical section, combine them into one bounded workflow that owns one lock for the duration.

Do not persist a workflow lock merely to reserve a worktree between independent Flue runs. Durable admission state should represent reservation; locks should protect active mutation.

### 4. [P1] Poll timestamps still count as meaningful watch changes

`src/modules/watches/service.ts` compares the full PR snapshot with `JSON.stringify`. Merged PR snapshots contain a check summary whose `checkedAt` is regenerated by `src/modules/github/checks.ts` on every fetch.

Ref watches use `meaningfulRefSnapshot`, but that helper includes the complete checks object and therefore also includes its refreshed observation timestamp.

Impact:

- unchanged merged PR watches and ref watches can be reported as changed on every poll;
- notifications reconcile repeatedly and may reset unread state;
- noisy basic-watch changes obscure the higher-value event-watermark delta path.

Recommended fix:

- define a shared semantic check fingerprint containing status, totals, conclusions, and relevant SHA/context facts;
- exclude `checkedAt`, `fetchedAt`, and other observation timestamps from change detection;
- persist observation timestamps separately for freshness and diagnostics.

### 5. [P1] Scheduling has two overlapping, pre-release implementations

The `schedules.json` implementation:

- `src/runtime-home/schemas.ts` accepts `cron` and `timezone`;
- `src/modules/config/mutations/schedules.ts` persists those fields;
- `src/modules/scheduler/service.ts` ignores them and compiles every schedule to `intervalSeconds`;
- a newly compiled job is immediately due and subsequent runs are calculated as `tickNow + interval`;
- `src/modules/scheduler/dispatch.ts` treats unknown job types as silent no-ops.

Routines:

- `src/modules/routines/service.ts` supports `interval`, `once`, and `cron`;
- cron is implemented by a custom minute-by-minute UTC scanner;
- the UI accepts cron expressions but does not label them as UTC or expose a timezone;
- only numeric values, numeric lists, and `*` are supported;
- day-of-month and day-of-week are matched with AND semantics, which differs from common Vixie cron behavior when both fields are restricted.

Impact:

- identical-looking schedule concepts behave differently depending on which subsystem owns them;
- `timezone` is a dead configuration field in one subsystem and unavailable in the other;
- a “morning briefing” is currently an interval from admission, not a reliable local wall-clock schedule, and will not naturally preserve local time across DST;
- custom cron parsing adds correctness and maintenance cost.

Recommended fix:

- delete `schedules.json`, its runtime path/bootstrap/health handling, config actions, blueprint action, and duplicate scheduler compilation path;
- retain one SQLite-backed scheduled-task table and one run-history table;
- define three validated task payloads: `poll-pr-watch`, `run-briefing`, and `run-agent-instruction`;
- define one discriminated trigger contract and use it for all three task kinds;
- use one tested Node cron implementation with explicit IANA timezone support for host timing; Flue's current Node schedule guidance names Croner as an example and does not prescribe its own cron engine;
- reject unsupported job/blueprint types instead of silently advancing them;
- expose typed creation actions for briefings and agent instructions; PR-watch task creation should remain an internal consequence of creating/enabling a watch.

Because the application has not shipped, do not add a migration layer, compatibility reader, deprecated alias, or dual-write period for the removed schedule format.

### 6. [P2] Autopilot concurrency is advisory rather than an atomic admission control

`src/modules/autopilot-policy/service.ts` reads active Flue observations and active/busy worktrees, then returns an allow/block decision. It does not reserve capacity in the same transaction.

Consequences:

- two concurrent admissions can both observe available capacity and proceed;
- per-repo counts miss triage runs and other active work without linked worktrees;
- `maxAutonomousJobs` and `maxActiveWorkflowRuns` are both compared against the same active-run count;
- local execution concurrency is a module-level counter and therefore process-local;
- same-PR worktree locks are atomic, but the global and per-repo limits are not.

Pending triage is also stored inside the scheduler job's nested `lastResult` JSON rather than as a queryable durable queue row. That is clever for retrying a small number of events, but it makes admission state hard to inspect, index, reconcile, prioritize, or migrate.

Recommended fix:

- claim an autopilot admission slot transactionally in app SQLite;
- make `(watch_id, event_fingerprint)` unique for idempotency;
- record repo, PR, mode, state, attempt count, next retry, workflow run id, and worktree/prepared-diff linkage;
- derive global/per-repo/same-PR usage from active admission rows;
- retain the PR lock as the final atomic mutation guard;
- remove one of the two global limits unless distinct semantics are defined.

### 7. [P2] “Approval required” is not satisfiable by the current push approval

`src/modules/autopilot-policy/service.ts` distinguishes hard blocks from approval-required diffs. The push workflow then requires both:

- `policy.approvalRequired === false`; and
- `preparedDiff.pushApprovalStatus === approved`.

An explicit prepared-diff approval therefore does not satisfy the policy's approval-required result. High-risk CI fixes are also retained before commit, with no typed approve-and-commit continuation. Even after manual commit, the same high-risk classification remains and the push policy gate still fails.

Recommended fix:

- model the policy decision as `deny | require-approval | allow`;
- bind approvals to the exact commit SHA, policy revision, purpose, and stage;
- let a matching approval satisfy `require-approval`;
- keep `deny` non-overridable;
- if some classes are intentionally manual-only, label them `manual-only` rather than offering an approval that cannot advance the state.

### 8. [P2] Planning and operator terminology are stale

`.plans/AUTOPILOT_LOOP_WIRING_PLAN.md` still says scheduler PR-watch jobs never touch event watermarks and triage never fires automatically. That is no longer true on current `main`.

At the same time, `src/modules/autopilot/state.ts` still describes the queue as placeholder adapters derived from other tables until workflow admission rows land. That is accurate, but it conflicts with Phase 19/20 being marked complete in `.plans/ROADMAP.md`.

Recommended fix:

- update the wiring plan to mark automatic delta-to-triage ignition complete;
- keep downstream triage-to-fix orchestration open;
- reopen or narrow the Phase 19 completion statement;
- call the current dashboard view “derived autopilot activity” until a durable admission queue exists.

## What is already strong

The following mechanics should be retained:

- separate Neondeck app state and Flue runtime state;
- scheduler tick lease plus observed-workflow admission lease;
- per-job scheduler failure isolation and attention notifications;
- advancing due tasks before external side effects to reduce duplicate workflow admission;
- event watermark baselining before triage;
- effective watch policy resolved server-side rather than trusted from watcher input;
- deterministic PR event ids and retry/supersede handling;
- managed worktrees as the mutation boundary;
- atomic PR/worktree lock acquisition;
- prepared-diff SHA binding, verification records, retained blocked worktrees, and force-push refusal;
- routines kill switch, minimum cadence, transactional claims, global cap, single-flight, stale-claim recovery, repeat limits, and auto-pause;
- deterministic notification source ids and recovery metadata.

## Recommended simplification

### A. One Node timing layer, with Flue owning scheduled execution

This design must be revalidated against the latest installed Flue version after the version prerequisite above. If a newer Flue release adds a Node timing adapter or durable schedule primitive, use it and reduce the Neondeck host layer accordingly. Do not retain custom timing, dispatch, persistence, or run-observation code that the selected Flue release now provides.

Use one canonical scheduled-task record with a discriminated, schema-validated payload:

```ts
type ScheduledTaskSpec =
  | { kind: 'poll-pr-watch'; watchId: string }
  | { kind: 'run-briefing'; briefingId: string }
  | {
      kind: 'run-agent-instruction';
      prompt: string;
      target:
        { kind: 'workflow' } | { kind: 'agent-session'; sessionId: string };
      repoId?: string;
      cwd?: string;
      skills: string[];
      delivery: DeliveryTarget;
    };
```

The task kind determines dispatch; arbitrary job-type strings should never reach the scheduler. The payload must be validated with Valibot when written and when read from SQLite.

Ownership stays simple:

- the watch service creates, updates, pauses, or removes its `poll-pr-watch` task;
- a typed briefing action creates or updates `run-briefing` tasks;
- the existing routine action surface creates `run-agent-instruction` tasks from natural-language requests;
- the host scheduler calculates due times and claims tasks transactionally;
- a PR poll runs deterministic application code and sends only meaningful changes into the Flue autopilot path;
- a briefing occurrence starts a bounded Flue workflow;
- an agent-instruction occurrence starts a bounded Flue workflow by default, or explicitly sends an event to a continuing Flue agent session when continuity is required;
- one task-run table records every attempt and terminal result.

PR polling remains deterministic infrastructure work. It should invoke agent reasoning only after the poll produces a meaningful semantic event. Briefings and independent scheduled instructions should enter bounded Flue workflows; a recurring workflow run must not remain open between occurrences. Continuing-agent delivery should use Flue's event/session model, not a permanently running scheduler workflow.

Use one shared trigger type:

Introduce one shared trigger type:

```ts
type AutomationTrigger =
  | { kind: 'interval'; everySeconds: number }
  | { kind: 'once'; at: string }
  | { kind: 'cron'; expression: string; timezone: string };
```

One service should own:

- validation;
- next-occurrence calculation;
- minimum spacing checks;
- timezone and DST behavior;
- human-readable descriptions;
- missed-run policy;
- test fixtures.

This trigger service is host infrastructure, not a replacement for Flue schedules. Its only responsibilities are determining when an occurrence is due and claiming it once. Flue owns the workflow/agent run identity, lifecycle events, history, and result for scheduled agent work.

Delete the generic schedule-blueprint abstraction. “Watch this PR,” “schedule a briefing,” and “run this instruction every 12 hours” should be typed product operations that produce a scheduled task, not stringly named presets in a second config system.

Recommended missed-run policy:

- interval: fixed delay from admission or completion, selected explicitly;
- cron: schedule the next wall-clock occurrence and coalesce missed occurrences to one run after restart;
- once: run once when due, then disable;
- never replay an unbounded backlog.

### B. One durable autopilot admission state machine

Add a first-class app-state table such as:

```text
autopilot_admissions
  id
  watch_id
  event_fingerprint
  repo_id
  pr_number
  mode
  state
  priority
  current_workflow
  current_run_id
  worktree_id
  prepared_diff_id
  attempt_count
  next_attempt_at
  last_error
  created_at
  updated_at

UNIQUE(watch_id, event_fingerprint)
```

Suggested states:

```text
observed
triage-admitted
triaged
prepare-admitted
prepared
fix-admitted
fix-prepared
policy-review
verify-admitted
verified
push-approval
push-admitted
pushed
comment-admitted
completed
blocked
superseded
```

Ownership should be explicit:

- watcher/scheduler: observe facts, compute semantic delta, insert or reconcile admission;
- coordinator: select the next state transition and atomically reserve capacity;
- Flue workflow: execute one bounded unit and return structured terminal facts;
- worktree lock: protect only active mutation;
- prepared diff: remain the source of truth for review/approval/verification/push facts;
- dashboard/TUI: read admissions directly, not reconstruct a queue from unrelated tables.

This removes the need to persist pending triage inside scheduler job-result JSON and gives retries, priority, observability, learning, and operator recovery one durable source of truth.

### C. One policy decision model

Replace overlapping booleans with an explicit result:

```ts
type PolicyDecision =
  | { decision: 'deny'; reasons: string[] }
  | { decision: 'require-approval'; approvalClass: string; reasons: string[] }
  | { decision: 'allow'; reasons: string[] };
```

Every approval should be bound to:

- repo and PR;
- worktree/prepared diff;
- exact commit SHA;
- policy revision/hash;
- operation (`commit`, `verify`, `push`, `comment`);
- approver surface and timestamp.

This makes it obvious which conditions are overridable and prevents a generic “push approved” flag from being confused with approval for high-risk content.

### D. Smaller configuration surface

- create a dedicated typed autopilot-policy mutation instead of hiding it under repo metadata;
- use one canonical autopilot mode vocabulary and remove the pre-release aliases;
- remove one of `maxAutonomousJobs` and `maxActiveWorkflowRuns` unless their counters differ;
- remove `schedules.json` and every read/write/action/UI/CLI reference to it;
- replace generic schedule blueprints with the three typed scheduled-task operations;
- reject unknown scheduled-task kinds at write and read boundaries;
- keep observation timestamps out of semantic state comparisons;
- archive or update stale plan documents once the implemented state changes.

1. Re-check npm dist-tags and upgrade the complete Flue package set to npm's latest published release; pin the selected prerelease versions, update the lockfile, and validate existing Flue entrypoints.
2. Re-read the installed release's schedule/workflow/agent/persistence APIs and adjust this architecture to remove anything Flue now owns.

4. Fix semantic watch comparison so unchanged checks are quiet.
5. Stop `preparePrWorktree` from retaining a default cross-workflow lock.
6. Add typed, confirmation-aware autopilot policy mutation and close the metadata bypass.
7. Replace `schedules.json`, blueprint scheduling, `jobs`, and `routines` with one SQLite scheduled-task/run model and one Node trigger engine that admits Flue workflows or agent events.
8. Add durable autopilot admissions with atomic concurrency claims.
9. Wire terminal triage results into the admission state machine and later bounded workflows.
10. Make approval-required policy satisfiable through SHA-bound approvals.

## Verification recommendations

When implementation begins, add focused coverage for:

- all `@flue/*` packages resolve to the intended compatible release family with no duplicate runtime/SDK versions;
- existing agent dispatch, workflow invocation, run observation, persistence, routing, and React client integration still work after the Flue upgrade;
- a source scan finds no compatibility actions, old memory scopes, autopilot policy aliases, `neon_sessions`, `legacyReason`, unjournaled-database upgrade path, Kilo environment-name fallback, or legacy MCP transport;
- fresh app databases are created exclusively from the new squashed baseline, while later forward migrations still apply normally;
- unchanged merged PR/ref checks with different `checkedAt` timestamps remain silent;
- cron next-run behavior in named timezones across DST boundaries;
- fresh setup creates no `schedules.json` file and exposes no obsolete schedule config actions;
- each of the three task kinds validates, persists, claims, dispatches, and records runs through the same scheduler;
- “every 12 hours, use CLI `xyz` to do `abc`” creates one agent-instruction task and one bounded Flue run per occurrence;
- watch creation produces exactly one internal PR-poll task, and an unchanged poll invokes no agent workflow;
- a local-time briefing remains at the requested wall-clock time across DST transitions;
- two simultaneous autopilot admissions cannot exceed global/per-repo limits;
- the same event fingerprint cannot create two admissions;
- prepare followed by fix succeeds using default inputs without waiting for a stale lock;
- increasing repo/watch autopilot authority requires explicit confirmation;
- approved high-risk changes advance only when the approval matches the current commit and policy;
- triage terminal results produce the correct next admission for every mode;
- restart recovery resumes blocked/failed admissions without replaying completed mutations.
