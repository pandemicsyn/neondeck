How the frontend works

targeting map used before delegation

The review scopes were chosen from this flow. The dashboard is not a route-heavy SPA; it is a config-driven plugin shell with two server-state transports and one focused PR review route.

→

1 · Root providers

main.tsx creates one TanStack Query client and one Flue SDK client mounted at /api/flue.

→

2 · Shell + config

App.tsx loads dashboard JSON, applies theme/density, classifies the measured deck, and renders regions.

→

3 · Plugin registry

Each active region mounts one typed plugin. Inactive tabs unmount; the registry itself eagerly imports every plugin.

→

4 · Server state

TanStack Query owns app reads/mutations; a singleton SSE hub multiplexes config, notification, session, and review events.

5 · Agent + diffs

@flue/react owns live agent sessions. Heavy PR/diff surfaces lazy-load Pierre React workers and renderers.

Diff lens. The working tree has no diff. The performance reviewer used 8c972c0^..HEAD instead: 20 frontend files and about 1,325 added lines across the PR workbench, Reviews backfill, multiplexed events/report overlays, and notification sound.

P1Large Pierre patches render without virtualization or a client cap

Impact and trigger

Opening a very large file or long wrapped patch can mount excessive DOM and syntax-highlight work, degrading scroll, resize, memory, and open-to-highlight latency. CSS overflow creates a scroll container but does not reduce mounted rows.

Recommendation after measurement

Benchmark Pierre React's native virtualizer and a clearly disclosed patch-size cap/truncation path. Measure DOM nodes, heap, long tasks, scroll frames, resize cost, and open-to-highlight first.

P1Review open can fan out patch requests across every unresolved-thread and draft path

Impact and trigger

Many unresolved threads or draft comments across large files create a burst of independent, uncancelled requests. Every response rebuilds maps, merged file arrays, and anchor indexes, increasing network, backend, CPU, and cache cost.

Recommendation after measurement

Separate active/adjacent display patches from non-visible anchor validation; pass AbortSignals and byte caps; cache anchors per patch identity; evaluate batched server validation.

P1Chat typing rerenders and reparses the full non-windowed Markdown timeline

Impact and trigger

Composer state shares a component with the complete durable timeline. Each keystroke remaps the timeline and revisits every Markdown message; fresh Markdown config props defeat cheap memo boundaries. Latency and GC scale with history length.

Recommendation after measurement

Profile 10/100/500-message sessions, then isolate composer state, stabilize Markdown configuration and callbacks, memoize message subtrees/timeline derivation, and only then evaluate paging or virtualization.

What Neondeck can learn from Hunk

comparative evidence
Hunk uses @pierre/diffs core 1.2.2 with OpenTUI; Neondeck uses Pierre React 1.2.12. The rendering implementations are not equivalent. Transfer the discipline: benchmarks, identity stability, and prefetch policy, not terminal-specific code.

Transferable patterns

  • Benchmark realistic tiers: Hunk includes 180×120 and 1,000-file/300-line fixtures plus a 50,000-line file.
  • Measure selected-file startup, next-file readiness, and adjacent readiness separately.
  • Track cold/warm first frame, windowed scroll ticks, interaction latency, and memory.
  • Use explicit file/row geometry and spacer preservation when windowing.
  • Keep callbacks stable at memoized section/row boundaries using latest-value refs and caches.

Neondeck already aligns

  • First and adjacent PR patch prefetch is already present.
  • Pierre worker configuration is module-scoped and stable.
  • Heavy review/diff surfaces have targeted lazy boundaries.
  • Artifact overlay aborts superseded/unmounted fetches.
  • App-domain EventSource traffic is multiplexed without replacing Flue realtime.
