How the frontend works

targeting map used before delegation

The review scopes were chosen from this flow. The dashboard is not a route-heavy SPA; it is a config-driven plugin shell with two server-state transports and one focused PR review route.

→

1 · Root providers

main.tsx creates one TanStack Query client and one Flue SDK client mounted at /api/flue.

→

2 · Shell + config

App.tsx loads dashboard JSON, applies theme/density, classifies the measured deck, and renders regions.

→

3 · Plugin registry

Each active region mounts one typed plugin. Inactive tabs unmount; the registry itself eagerly imports every plugin.

→

4 · Server state

TanStack Query owns app reads/mutations; a singleton SSE hub multiplexes config, notification, session, and review events.

5 · Agent + diffs

@flue/react owns live agent sessions. Heavy PR/diff surfaces lazy-load Pierre React workers and renderers.

Diff lens. The working tree has no diff. The performance reviewer used 8c972c0^..HEAD instead: 20 frontend files and about 1,325 added lines across the PR workbench, Reviews backfill, multiplexed events/report overlays, and notification sound.

Impact and trigger

Opening a very large file or long wrapped patch could mount excessive DOM and syntax-highlight work, degrading scroll, resize, memory, and open-to-highlight latency. CSS overflow created a scroll container but did not reduce mounted rows. This original condition is retained for audit history.

Confirmed, compared, then fixed. The retained 10,000-line baseline mounted about 40,099 nodes and accumulated about 400 ms of long tasks. At 10,000 rows it reduced ready time 76.9%, DOM 98.2%, and heap 43.4%, removed 405 ms of long tasks, and held scroll p95 at 12.8 ms.

Patches below 2,000 changed rows retain the lower-overhead PatchDiff path; larger patches use Pierre CodeView with calibrated line and header geometry, the worker pool, theme and wrap options, selection mapping, and review annotations. The production-shaped harness verified selected deletion rows and all 50 annotations. Ordinary-patch measurements confirmed why the threshold remains necessary.

P1Review open can fan out patch requests across every unresolved-thread and draft path COMPLETED

Impact and trigger

Many unresolved threads or draft comments across large files create a burst of independent requests. Every response rebuilds maps, merged file arrays, and anchor indexes, increasing network, backend, CPU, and cache cost. AbortSignals are now passed after the correctness PR, so the original “uncancelled” detail is retained as historical context but no longer applies.

Measurement result

Confirmed and refined with a real trace. Open Kilo-Org/kilocode#12204 had 1,019 files, 47 threads, and only two unresolved paths, so it did not reproduce the modeled 50-path maximum. It exposed two stronger costs. First, every scripted browser sample started two review-thread requests because the optimistic and authoritative PR records had different updatedAt-based query keys; one run transferred both 381 kB payloads, and complete traces abandoned then reissued two unresolved-path reads. Second, every local patch request recomputed metadata for all 1,019 files before the path diff. A sequential patch took 643 ms median while four real active/neighbor/unresolved paths took 1,952 ms concurrently. Production medians were 869 ms to tree, 1,971 ms to first patch, and 2,183 ms to threads.

Recommendation after measurement

Do not broaden the patch cap yet. First prevent optimistic-to-authoritative review-thread key churn while retaining explicit invalidation. Then cache or single-flight immutable local metadata by repo/merge-base/head and remeasure. Only if the active patch still loses to background work should neighbor and unresolved-path prefetch wait for it to settle. Cold local object fetch also missed its target at 4.98 seconds, but should be separated from repeat-path work before changing fetch behavior or the budget.

Completed remediation

Review threads now use a stable PR-scoped query key while mutations and later authoritative activity changes explicitly refresh it. Metadata-only local diffs use a bounded 10-minute, eight-revision LRU keyed by repo path, merge-base, and head; concurrent misses single-flight and failed reads are not cached. Patch text remains on demand. The active patch now runs separately; neighbor, draft-comment, and unresolved-thread reads wait until it settles and remain cancellable on selection changes. Deferred draft anchors stay unknown without masking removed-file staleness. On the retained target, browser first patch improved from 1,195 to 798 ms and passed all three samples, patch reads fell from six to four, the last API response moved from 2,782 to 2,021 ms, and every sample retained one thread request and zero abandoned patches.

Retained follow-up

The original broad many-path risk remains in this audit history, but it no longer competes with the initial selected patch. The one-time 4.98-second cold object fetch, 612 ms tree median, and 1,724 ms thread median remain separate measurement decisions rather than blockers for this completed item.

What Neondeck can learn from Hunk

comparative evidence
Hunk uses @pierre/diffs core 1.2.2 with OpenTUI; Neondeck uses Pierre React 1.2.12. The rendering implementations are not equivalent. Transfer the discipline: benchmarks, identity stability, and prefetch policy, not terminal-specific code.

Transferable patterns

  • Benchmark realistic tiers: Hunk includes 180×120 and 1,000-file/300-line fixtures plus a 50,000-line file.
  • Measure selected-file startup, next-file readiness, and adjacent readiness separately.
  • Track cold/warm first frame, windowed scroll ticks, interaction latency, and memory.
  • Use explicit file/row geometry and spacer preservation when windowing.
  • Keep callbacks stable at memoized section/row boundaries using latest-value refs and caches.

Neondeck already aligns

  • First and adjacent PR patch prefetch is already present.
  • Pierre worker configuration is module-scoped and stable.
  • Heavy review/diff surfaces have targeted lazy boundaries.
  • Artifact overlay aborts superseded/unmounted fetches.
  • App-domain EventSource traffic is multiplexed without replacing Flue realtime.
